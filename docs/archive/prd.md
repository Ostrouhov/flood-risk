# PRD: Прототип ИНС-оценки подверженности затоплению

> **Документ предназначен для исполнителя (другой Claude Code), который будет собирать прототип по этим требованиям.**
> Источники истины верхнего уровня: [discovery.md](discovery.md) (постановка), [concept.md](concept.md) (скоуп MVP).
> Эта PRD — детализация скоупа до атомарных тестируемых требований.
>
> **Версия:** 0.1
> **Дата:** 2026-05-29
> **Статус:** черновик, требует прохода по open questions перед стартом ТЗ.

---

## 1. TL;DR

Прототип — веб-сервис, который по bbox городского района и выбранному метео-сценарию возвращает растровую карту вероятности затопления, построенную обученной U-Net моделью на открытых геоданных. Демо включает воспроизводимый пайплайн данных по одной пилотной территории, два обученных модели (U-Net и табличный бейзлайн) с зафиксированными метриками, REST API инференса, веб-карту с тремя сценариями и экспортом, а также карту важности признаков для объяснимости. Целевая аудитория демо — научный руководитель и комиссия защиты диссертации; коммерческого «клиента» нет, успех = принятие работы.

---

## 2. Проблема и аудитория

### 2.1 Проблема

Классические гидродинамические модели (HEC-RAS, MIKE FLOOD) требуют закрытых ведомственных данных и ручной калибровки гидрологом → переоценка рисков по российским городам организационно и экономически нецелесообразна. Параллельно открытые геоданные (DEM, Sentinel-1, ESA WorldCover, OSM, ERA5) достигли качества, при котором data-driven подход даёт оценку «достаточно хорошо» при радикально меньших требованиях к входу.

### 2.2 Аудитория прототипа

| Персона | Что делает с прототипом | Канал |
|---|---|---|
| **Первичная: специалист отдела градостроительства муниципалитета** (синтетическая) | Открывает карту, переключает сценарий, выгружает GeoTIFF + PDF для проектной документации | Веб-UI |
| **Вторичная: исследователь (автор работы, научный руководитель)** | Переобучает модель, прогоняет инференс на новых bbox, сравнивает архитектуры | CLI / Jupyter |
| **Аудитория защиты (комиссия, оппонент)** | Смотрит живую демонстрацию на проекторе; задаёт вопросы про объяснимость | Веб-UI + ответы автора |

> Реального клиента нет. Все продуктовые решения принимаются под первичную персону как референс (зафиксировано в discovery §3).

## 3. Цели и не-цели

### 3.1 Цели прототипа

| # | Цель | Связь с discovery |
|---|---|---|
| G-1 | Получить карту вероятности затопления для пилотной территории через веб-интерфейс за единицы секунд | §9.2.2, §9.2.3 |
| G-2 | Показать статистически значимое превосходство U-Net над табличным бейзлайном по IoU / F1 / PR-AUC на geographic hold-out | §9.1.1 |

| G-4 | Обеспечить минимальную объяснимость предсказаний (минимум один метод) для защиты от критики на защите | §6.5, §10 |
| G-5 | Дать вторичной персоне воспроизводимый стенд: CLI-инференс и переобучение через конфиги | §3, §4 U4 |

### 3.2 Не-цели (anti-goals)

Перечислены явно, чтобы исполнитель не расширял скоуп:

- ❌ Оперативное оповещение, real-time API, push-уведомления.
- ❌ Прогноз метеоусловий (сценарии подаются как фиксированные пресеты).
- ❌ Гидродинамика во времени (распространение волны паводка).
- ❌ Оценка ущерба в денежном выражении.
- ❌ Подмена государственной экспертизы зон затопления.
- ❌ Мобильное приложение, оффлайн-режим, PWA.
- ❌ Мультитенантность, аутентификация, RBAC, биллинг.
- ❌ Прод-деплой, нагрузочное тестирование, мониторинг SLA.
- ❌ Дообучение под новый регион через UI.
- ❌ Ансамбли моделей, калибровка неопределённости на пиксель.
- ❌ Интеграция с реальной ливневой канализацией и закрытыми ведомственными данными.
- ❌ Платные коммерческие спутниковые снимки.

## 4. Метрики успеха демо

Демо считается принятым, если на живой демонстрации (защита или внутренний show-and-tell) выполнены **все** пункты ниже. Каждый пункт — измеримый, без формулировок «должно быть удобно».

| # | Метрика | Способ проверки | Порог приёмки |
|---|---|---|---|
| M-1 | Воспроизводимая сборка | На чистой машине из фиксированного коммита выполняются `docker compose up` + `make data` + `make train` | ≤ 1 рабочего дня (8 часов) до получения карты риска |
| M-2 | Качество модели | Метрики из MLflow / W&B run’а на geographic hold-out | U-Net превосходит бейзлайн по IoU **и** F1 **и** PR-AUC; значимость подтверждена bootstrap-CI (95%) с непересекающимися интервалами либо p-value < 0.05 |
| M-3 | Скорость инференса | `curl -X POST /predict` с фиксированным bbox пилотной территории, замер p95 на 20 запросах | p95 ≤ 5 сек на CPU (8 ядер), ≤ 1 сек на GPU |
| M-4 | UI-сценарии | Ручной прогон U1 (просмотр), U2 (переключение 3 сценариев), U3 (экспорт GeoTIFF + GeoJSON + PDF) на пилотной территории | 0 ошибок, 0 фризов > 2 сек на действие пользователя |
| M-5 | Объяснимость | Для выбранного тайла открывается карта важности признаков с легендой | Карта отрисована; метод и его ограничения описаны в README одним абзацем + 1 пример |
| M-6 | Документация | Внешний человек (научный руководитель) по README поднимает систему и получает карту | Время прохождения README ≤ 1 день; ноль обращений «у меня не запустилось» из-за отсутствующих шагов |

## 5. User flows

Для каждой IN-фичи из concept §2.1 — пошаговый сценарий пользователя на уровне «что нажал → что увидел».

### 5.1 UF-1 (IN-1): Воспроизводимая сборка датасета — исследователь

1. Исследователь клонирует репозиторий и делает `git checkout <release-tag>`.
2. Запускает `docker compose up -d` → поднимаются PostgreSQL+PostGIS и сервисный контейнер.
3. Запускает `make data BBOX=<пилотная-территория>` → скрипт скачивает Copernicus DEM, ESA WorldCover, OSM-выгрузку, ERA5 за заданный период, Sentinel-1 за окно исторического события.
4. Скрипт строит маску постоянной воды (OSM + JRC GSW) и отделяет её от событий затопления.
5. Скрипт нарезает тайлы фиксированного размера и записывает в `data/processed/v1/`.
6. В `data/manifest.yaml` появляются контрольные суммы тайлов, версии источников, сид нарезки.
7. Исследователь сравнивает `manifest.yaml` с эталонным из репозитория → совпадает по контрольным суммам.

**Где может сломаться:** обновление внешних источников (Copernicus, ESA) → manifest расходится. См. OQ-3.

### 5.2 UF-2 (IN-2): Обучение и фиксация метрик — автор работы

1. Автор выполняет `make train MODEL=unet CONFIG=configs/unet_v1.yaml`.
2. PyTorch Lightning логирует метрики IoU, F1, ROC-AUC, PR-AUC, Brier score по эпохам в MLflow.
3. По завершении в `models/unet/v1/` лежат чекпоинт + конфиг + лог метрик.
4. Аналогично `make train MODEL=baseline CONFIG=configs/baseline_v1.yaml` для RF/логистики.
5. `make eval` строит таблицу сравнения на test-split (geographic hold-out по bbox) с bootstrap-CI и/или p-value.
6. Результат сохраняется в `reports/comparison_v1.md` и пуш в репозиторий.

**Где может сломаться:** недостаточно исторических событий → переобучение. См. discovery §10 и OQ-2.

### 5.3 UF-3 (IN-3): Запрос инференса — фронтенд (и вторичная персона через CLI)

1. Клиент шлёт `POST /predict` с JSON: `{bbox: [west, south, east, north], scenario_id: "p10y", model_version: "unet-v1"}`.
2. Бэкенд валидирует bbox (попадание в зону покрытия модели), формирует входной растр признаков для тайла.
3. Бэкенд прогоняет инференс на загруженной модели (модель держится в памяти, не перезагружается на каждый запрос).
4. Бэкенд возвращает: URL на GeoTIFF предсказания + JSON с агрегатами (доля площади с p>0.5, p>0.8, средняя вероятность по bbox) + метаданные запуска.
5. p95 latency ≤ 5 сек на CPU. Логи запроса попадают в stdout контейнера.

**Где может сломаться:** bbox вне зоны покрытия → 422 с понятной ошибкой. См. FR-7.

### 5.4 UF-4 (IN-4): Работа специалиста муниципалитета с картой

1. Пользователь открывает в браузере `http://localhost:<port>/`.
2. Видит MapLibre-карту, центрированную на пилотной территории, и панель управления слева: селектор сценария (radio: «обычный год» / «1 раз в 10 лет» / «1 раз в 100 лет»), кнопка «Экспорт», селектор модели (по умолчанию — U-Net).
3. Пользователь выбирает сценарий «1 раз в 100 лет» → фронт шлёт `/predict`, через ≤ 5 сек растровый оверлей перерисовывается.
4. Видит легенду шкалы вероятности (например, 5 градаций цвета 0–1) и агрегаты по текущему bbox (доля затопляемой площади).
5. Нажимает «Экспорт» → получает архив с тремя файлами: `prediction.tif`, `aggregates.geojson`, `report.pdf`. В PDF — карта-скриншот, легенда, метаданные запуска (модель, версия датасета, сценарий, bbox, дата).
6. Сценарии U1 (просмотр), U2 (переключение), U3 (экспорт) проходят без ошибок.

**Где может сломаться:** PDF-генерация на CI без графической подсистемы — нужен headless-режим. См. OQ-5.

### 5.5 UF-5 (IN-5): Объяснимость для одного тайла

1. На карте пользователь кликает по точке интереса → активируется режим «Объяснение».
2. Фронт шлёт `POST /explain` с координатами клика и `scenario_id`.
3. Бэкенд рассчитывает карту важности признаков (метод — см. OQ-4) для тайла, содержащего точку.
4. Возвращает массив растров «по одному на признак» + JSON с ранжированием признаков по среднему вкладу.
5. UI отрисовывает мини-панель с топ-5 признаками и переключателем растровых слоёв важности.
6. В README — описание метода и его ограничений с примером.

**Где может сломаться:** integrated gradients на U-Net медленный для большого тайла → urезаем тайл до окрестности клика. См. NFR-3 и OQ-4.

## 6. Функциональные требования

> Атомарные, тестируемые. Каждое FR обязательно имеет **признак готовности (Done)** — что именно проверяем, чтобы сказать «работает».

### 6.1 Data layer

#### FR-1. Скрипт загрузки сырых данных по bbox

- **Что:** CLI-команда `make data BBOX=...` (или `python -m floodrisk.data.fetch`) скачивает: Copernicus DEM GLO-30, ESA WorldCover, OSM-выгрузку (реки/водоёмы), ERA5/ERA5-Land за заданное окно, Sentinel-1 GRD за окно исторического события.
- **Done:** на чистом контейнере команда отрабатывает без ручных шагов; в `data/raw/` появляются все слои с фиксированными именами; команда идемпотентна (повторный запуск не качает повторно при наличии файла и совпадении контрольной суммы).
- **Тест:** integration-тест, запускающий `make data` на тестовом мини-bbox (~10×10 км), проверяет наличие выходных файлов и их непустоту.

#### FR-2. Препроцессинг и нарезка тайлов

- **Что:** скрипт строит производные признаки (уклон, экспозиция, кривизна, TWI, расстояние до водотока, GHSL urbanization), приводит все слои к единой проекции и разрешению (целевое — 30 м, EPSG — см. OQ-1), нарезает на тайлы фиксированного размера (целевое — 256×256, см. OQ-1).
- **Done:** `data/processed/v1/tiles/` содержит .tif-тайлы; `data/processed/v1/index.parquet` — таблицу с координатами тайла, метаданными, split-меткой (train/val/test); все тайлы имеют идентичную геометрию (одна и та же проекция, разрешение, размер).
- **Тест:** unit-тест проверяет: (а) одинаковый shape и CRS у случайной выборки тайлов; (б) непересечение train/val/test по bbox.

#### FR-3. Разметка целевой переменной по Sentinel-1

- **Что:** скрипт детектирует затопленные пиксели на снимках Sentinel-1 для исторических событий (методика — см. OQ-2); вычитает маску постоянной воды (OSM + JRC Global Surface Water); сохраняет бинарные маски «затоплено/нет» на тайл.
- **Done:** для каждого исторического события в `data/processed/v1/labels/<event_id>/` лежат бинарные маски по тайлам; в `data/manifest.yaml` зафиксирован метод детекции, пороги, версия маски постоянной воды.
- **Тест:** на одном эталонном событии маска совпадает с ручной валидацией (метрика — площадь пересечения; порог — см. OQ-2).

#### FR-4. Манифест и контрольные суммы

- **Что:** `data/manifest.yaml` содержит: версии всех источников (URL + дата выгрузки), сиды нарезки, контрольные суммы (sha256) всех тайлов, параметры препроцессинга.
- **Done:** манифест создаётся автоматически; команда `make verify-data` проверяет соответствие текущих файлов манифесту и завершается с exit code 0 при совпадении.

### 6.2 ML layer

#### FR-5. Обучение U-Net

- **Что:** `make train MODEL=unet CONFIG=configs/unet_v1.yaml` обучает U-Net (segmentation-models-pytorch) на тайлах из FR-2 с метками из FR-3.
- **Done:** в `models/unet/<run_id>/` лежат: чекпоинт `best.ckpt`, конфиг, лог метрик за эпохи (MLflow или W&B); финальные метрики IoU, F1, ROC-AUC, PR-AUC, Brier score на val и test.
- **Тест:** smoke-тест: 1 эпоха на мини-датасете → метрики записываются, чекпоинт сохраняется.

#### FR-6. Обучение табличного бейзлайна

- **Что:** `make train MODEL=baseline` обучает Random Forest (или логистическую регрессию) на тех же признаках, попиксельно, на тех же train/val/test splits.
- **Done:** в `models/baseline/<run_id>/` лежит сериализованная модель (joblib/pickle), конфиг, метрики на тех же выборках.
- **Тест:** аналогично FR-5.

#### FR-7. Сравнение и фиксация научной метрики

- **Что:** `make eval` строит таблицу сравнения U-Net vs baseline на test-split с bootstrap-CI (95%) для IoU / F1 / PR-AUC; результат — `reports/comparison_v1.md`.
- **Done:** отчёт содержит: таблицу метрик, bootstrap-CI и/или p-value (метод — см. OQ-6), pass/fail-вывод по M-2.

#### FR-8. CLI-инференс (для вторичной персоны)

- **Что:** `python -m floodrisk.infer --bbox=... --scenario=p100y --model=unet-v1 --out=prediction.tif` производит инференс без поднятия API.
- **Done:** команда отрабатывает на чистом контейнере; на выходе — GeoTIFF предсказания и JSON с агрегатами.

### 6.3 Inference service

#### FR-9. REST-эндпоинт `/predict`

- **Что:** `POST /predict` с JSON `{bbox: [w,s,e,n], scenario_id: str, model_version: str}` возвращает `{prediction_url: str, aggregates: {...}, metadata: {model_version, dataset_version, scenario_id, bbox, run_timestamp}}`. OpenAPI-схема сгенерирована автоматически.
- **Done:** Swagger UI доступен по `/docs`; запрос на фиксированном bbox пилотной территории возвращает 200 + валидный GeoTIFF по `prediction_url`; smoke-test в CI проходит.

#### FR-10. REST-эндпоинт `/explain`

- **Что:** `POST /explain` с JSON `{lat: float, lon: float, scenario_id: str, model_version: str}` возвращает карту важности признаков (метод — см. OQ-4) для тайла вокруг точки + ранжирование признаков.
- **Done:** Swagger описание есть; запрос отдаёт массив растров (по одному на топ-N признаков) + JSON с ранжированием; latency — см. NFR-3.

#### FR-11. REST-эндпоинт `/scenarios`

- **Что:** `GET /scenarios` возвращает список преднастроенных сценариев с человекочитаемыми описаниями: `[{id: "p1y", name: "Обычный год", ...}, {id: "p10y", ...}, {id: "p100y", ...}]`. Точные параметры — см. OQ-7.
- **Done:** эндпоинт возвращает минимум 3 сценария; описания берутся из конфига, не зашиты в код.

#### FR-12. Валидация bbox и обработка ошибок

- **Что:** при bbox вне зоны покрытия модели — HTTP 422 с body `{error: "bbox_out_of_coverage", coverage: [w,s,e,n]}`. При неизвестном `scenario_id` или `model_version` — 404 с пояснением.
- **Done:** unit-тесты на каждый кейс ошибки.

#### FR-13. Загрузка модели в память при старте

- **Что:** контейнер загружает чекпоинт U-Net и бейзлайн в память при старте; повторные запросы используют резидентную модель.
- **Done:** замер `/predict` показывает, что от 2-го запроса и далее latency стабильна (нет «холодного старта» на каждый запрос).

### 6.4 Frontend

#### FR-14. MapLibre-карта с растровым оверлеем

- **Что:** SPA на React + MapLibre GL; при загрузке центрируется на bbox пилотной территории; над базовой картой отрисовывается полупрозрачный растровый слой предсказания.
- **Done:** растр виден поверх базовой карты; масштабирование/панорамирование работает без артефактов; цветовая шкала соответствует легенде.

#### FR-15. Селектор сценария

- **Что:** UI-элемент (radio или dropdown) с минимум 3 сценариями из `/scenarios`; смена сценария триггерит `/predict` и обновляет оверлей.
- **Done:** переключение между «p1y / p10y / p100y» показывает визуально разные карты; во время загрузки — спиннер или скелетон.

#### FR-16. Легенда и агрегаты

- **Что:** слева/справа от карты — панель: легенда шкалы вероятности (минимум 5 градаций), агрегаты по текущему bbox (доля площади с p>0.5, p>0.8, средняя вероятность).
- **Done:** при смене сценария агрегаты обновляются согласованно с растром.

#### FR-17. Экспорт

## 7. Нефункциональные требования

> Уровень — демо: 1 пользователь, объёмы — см. ниже. Никакого прод-SLA не закладываем.

| # | Категория | Требование | Способ проверки |
|---|---|---|---|
| NFR-1 | Latency `/predict` | p95 ≤ 5 сек на CPU (8 ядер), p95 ≤ 1 сек на GPU, на тайле пилотной территории | Замер на 20 последовательных запросах с фиксированным bbox |
| NFR-2 | Latency UI | От клика на смену сценария до перерисовки оверлея — p95 ≤ 6 сек на CPU | Ручной замер с секундомером на пилотной территории |
| NFR-3 | Latency `/explain` | p95 ≤ 15 сек на CPU для окрестности точки (size — см. OQ-4) | Замер на 10 запросах |
| NFR-4 | Объём данных демо | Пилотная территория — порядка нескольких тысяч км² (см. discovery §5.3); датасет v1 — порядка **1k тайлов** 256×256 (точное число фиксируется после нарезки) | Подсчёт строк в `data/processed/v1/index.parquet` |
| NFR-5 | Нагрузка | 1 одновременный пользователь; конкурентность не закладываем | Не тестируем нагрузку выше; gunicorn — 1 worker |
| NFR-6 | Размер артефактов | Чекпоинт U-Net + бейзлайн в сумме ≤ 500 МБ; репозиторий без LFS | Проверка `ls -lh models/` |
| NFR-7 | Воспроизводимость | Все запуски тренировки используют фиксированные сиды (PyTorch, NumPy, Python); requirements зафиксированы (requirements.lock или poetry.lock) | На двух прогонах с тем же сидом метрики совпадают в пределах документированной дисперсии (порог — см. OQ-8) |
| NFR-8 | Браузеры | Поддерживаются последние стабильные версии Chrome и Firefox на десктопе | Ручной прогон UF-4 в обоих браузерах |
| NFR-9 | Безопасность | API доступен только локально (bind 127.0.0.1); никаких секретов в коде; токены для скачивания данных — через `.env` | Code review + smoke-test |
| NFR-10 | Логирование | Backend пишет каждый запрос с bbox, сценарием, latency, размером ответа в stdout контейнера | Чтение `docker compose logs backend` |

## 8. Данные

### 8.1 Сущности

| Сущность | Где живёт | Формат |
|---|---|---|
| **Сырые слои** (DEM, WorldCover, OSM-выгрузка, ERA5, Sentinel-1) | `data/raw/<source>/` | GeoTIFF / GeoPackage / NetCDF (по источнику) |
| **Производные растры** (уклон, TWI, расстояние до водотока, …) | `data/processed/v1/features/` | GeoTIFF, 30 м, единая проекция (см. OQ-1) |
| **Тайлы признаков** | `data/processed/v1/tiles/<tile_id>.tif` | Многоканальный GeoTIFF, размер по OQ-1 |
| **Тайлы меток затопления** | `data/processed/v1/labels/<event_id>/<tile_id>.tif` | Бинарный GeoTIFF |
| **Индекс тайлов** | `data/processed/v1/index.parquet` | Parquet: `tile_id`, `bbox`, `event_ids`, `split` |
| **Манифест датасета** | `data/manifest.yaml` | YAML: источники, версии, сиды, контрольные суммы |
| **Чекпоинты моделей** | `models/<model>/<run_id>/` | PyTorch `.ckpt` / joblib `.pkl` |
| **Метрики экспериментов** | MLflow tracking (или W&B) + дублирование в `reports/` | MLflow store, MD-таблицы |
| **Запросы инференса** (логи) | stdout контейнера | JSON-lines |
| **Векторные слои UI** (границы районов, водотоки для подсветки) | PostGIS | Таблицы PostGIS |
| **PDF-отчёт экспорта** | Генерируется по запросу, не хранится | PDF |

### 8.2 Источники

Полный перечень — discovery §5.1, §5.2; ниже — выжимка с конкретикой:

| Слой | Источник | Канал доступа |
|---|---|---|
| DEM | Copernicus DEM GLO-30 | Открытое скачивание (AWS Open Data / Copernicus DataSpace) |
| Землепокрытие | ESA WorldCover 2021 | Открытое скачивание |
| Гидрография | OpenStreetMap (Geofabrik выгрузки) + HydroSHEDS | HTTP |
| Урбанизированность | GHSL | Открытое скачивание |
| Почвы | SoilGrids (ISRIC) | WCS / прямые тайлы |
| Осадки | ERA5 / ERA5-Land | CDS API (требует токен — `.env`) |
| Sentinel-1 (метки) | Copernicus DataSpace | API (требует токен — `.env`) |
| Маска постоянной воды | OSM + JRC Global Surface Water | Открытое скачивание |

### 8.3 Сценарии метео (входы модели)

Идентификаторы: `p1y`, `p10y`, `p100y`. Точные параметры (какие именно поля ERA5, за какое окно, какие квантили) — **открытый вопрос OQ-7**. В коде фиксируется набор сценариев как YAML-конфиг, чтобы менять без правок логики.

### 8.4 Что НЕ используем

Дублируется явно, чтобы исполнитель не «расширил источники» в процессе:
- Закрытые ведомственные данные (карты ливневой канализации, акты МЧС, ведомственные DEM).
- Платные коммерческие спутники.
- Real-time API сторонних сервисов.

## 10. Out of scope

> Полная копия из `concept.md` §2.2, чтобы было под рукой при ревью PRD и при добавлении новых требований не было соблазна «а давайте ещё».

| Что режем | Почему режем (со ссылкой на discovery) |
|---|---|
| Дообучение модели под новый регион через UI | §8.2 — вынесено в развитие; превращает прототип в платформу и ломает фокус «один регион — одна обученная модель». |
| Прогноз во времени / динамика паводка | §4 anti-goal: задача не про распространение волны, а про подверженность; смена постановки требует другой архитектуры и других данных. |
| Оперативное оповещение и интеграция с системами экстренного реагирования | §4 anti-goal; зона ответственности государственных систем, юридически и архитектурно вне нашего контура. |
| Оценка ущерба в денежном выражении | §4 anti-goal; требует данных о стоимости активов, которые сознательно не используются (§5.4). |
| Мультитенантность, аутентификация, ролевая модель | §4 и §8.2; прототип демонстрируется на защите, единственный пользователь — докладчик, auth — чистый overhead. |
| Мобильное приложение и оффлайн-режим | §4 anti-goal; первичная персона работает за десктопом с ГИС-инструментами (§3), мобильный сценарий не покрыт. |
| Интеграция с реальной ливневой канализацией | §8.2; требует закрытых ведомственных данных, что противоречит §5.4. |
| Ансамбли моделей и количественная оценка неопределённости | §8.2; ценно научно, но удваивает время на ML-фазу и не нужно для базового сравнения с бейзлайном (§9.1.1). |
| Развёртывание в продуктивной инфраструктуре, нагрузочное тестирование | §8.2; цель — Docker Compose на одной машине (§7.2), прод-нагрузки нерелевантны для защиты. |
| Real-time данные через сторонние API | §5.4 — явный запрет; ломает воспроизводимость, которая в работе важнее свежести данных. |
