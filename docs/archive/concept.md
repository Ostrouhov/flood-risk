# Concept: MVP-скоуп прототипа ИНС-оценки подверженности затоплению

> **Источник истины верхнего уровня:** [discovery.md](discovery.md).
> Этот документ фиксирует **скоуп защищаемого MVP** и обоснование каждой включённой/исключённой фичи через метрики и решения discovery.
> Остальные обязательные разделы концепта (научная новизна, обзор ближайших аналогов из литературы, окончательный выбор пилотной территории) — см. §5 ниже, заполняются в следующих итерациях и не блокируют переход к ТЗ по скоупу MVP.
>
> **Условные обозначения:**
> - `РЕШЕНИЕ:` — зафиксированный выбор, не пересматривается без явной ревизии.
> - `К ПРОРАБОТКЕ:` — открытый методический/инженерный вопрос.
>
> **Версия:** 0.1
> **Дата:** 2026-05-29

---

## 1. Продукт в одной строке

Веб-сервис, который по открытым геоданным и выбранному метео-сценарию строит карту вероятности затопления городского района и объясняет, какие признаки определили оценку.

---

## 2. MVP-скоуп

> РЕШЕНИЕ: защищаемый прототип = ровно 5 фич ниже. Любая дополнительная функциональность из бэклога подключается только после прохождения всех 5 признаков готовности.

### 2.1 IN — must-have для демо

#### IN-1. Воспроизводимый пайплайн данных по одной пилотной территории

- **Обоснование (метрика discovery):** §9.2.1 «любой человек по README ≤ 1 день поднимает систему» + §9.2.4 «пайплайн переигрывается из исходников и даёт метрики в пределах задокументированной дисперсии».
- **User story:** Как исследователь / научный руководитель, я хочу одной командой пересобрать датасет из открытых источников по фиксированной территории, чтобы убедиться в воспроизводимости результата.
- **Признак готовности:**
  - `make data` (или эквивалент) на чистой машине из фиксированного коммита собирает датасет v1 с воспроизводимыми контрольными суммами тайлов.
  - Маска постоянной воды (OSM / JRC Global Surface Water) отделена от событий затопления — закрывает риск §10 «Sentinel-1 шумная разметка».
  - Версия датасета, версии источников и сиды зафиксированы в манифесте.
- **Риск / неопределённость:** **8/10** — самый рисковый блок (§5 явно помечен); методика автоматической разметки по Sentinel-1 ещё `К ПРОРАБОТКЕ` (см. discovery §11.2); зависимость от GDAL/Docker (риск §10).

#### IN-2. U-Net + табличный бейзлайн с метриками на geographic hold-out

- **Обоснование (метрика discovery):** §9.1.1 «статистически значимое превосходство ИНС над бейзлайном по IoU/F1/PR-AUC» + §9.1.2 «обобщаемость на территориях, не входивших в обучение».
- **User story:** Как автор работы, я хочу обучить U-Net и бейзлайн на одинаковых признаках и зафиксировать метрики на удерживаемой по географии выборке, чтобы обосновать научную ценность ИНС-подхода.
- **Признак готовности:**
  - В репозитории — зафиксированные чекпоинты обеих моделей и YAML-конфиги экспериментов.
  - Отчёт MLflow / W&B с IoU, F1, ROC-AUC, PR-AUC, Brier score.
  - Разделение train/val/test по bbox (не по пикселям) — закрывает риск утечки.
  - Таблица сравнения U-Net vs бейзлайн с p-value или bootstrap-CI.
- **Риск / неопределённость:** **6/10** — архитектура устоявшаяся (discovery §6.2), но риск §10 «модель угадывает по очевидным признакам» реальный; статистически значимое превышение бейзлайна не гарантировано до эксперимента.

#### IN-3. Inference REST API (bbox + сценарий → растр + агрегаты)

- **Обоснование (метрика discovery):** §9.2.2 «инференс по тайлу — единицы секунд на CPU, доли секунды на GPU».
- **User story:** Как фронтенд (и как вторичная персона через CLI), я хочу отправить bbox и идентификатор метео-сценария и получить GeoTIFF предсказания + JSON со статистикой по зоне, чтобы единообразно питать карту и экспорт.
- **Признак готовности:**
  - FastAPI-эндпоинт `/predict` с задокументированной OpenAPI-схемой.
  - p95 latency ≤ 5 сек на CPU для тайла пилотной территории.
  - Smoke-тест в CI на фиксированном bbox и сценарии.
  - Контракт ответа (поля растра, агрегатов, метаданных запуска) зафиксирован в `tz.md` после написания.
- **Риск / неопределённость:** **3/10** — стандартная инженерия, стек зафиксирован (discovery §7.2).

#### IN-4. Веб-карта: U1 + U2 + U3 (просмотр / сравнение сценариев / экспорт)

- **Обоснование (метрика discovery):** §9.2.3 «UI отрабатывает сценарии U1–U3 без ошибок» + §4 фиксация скоупа фронта.
- **User story:** Как специалист отдела градостроительства, я хочу выбрать район, переключаться между преднастроенными сценариями («обычный год / 1 раз в 10 лет / 1 раз в 100 лет») и выгрузить карту в GeoTIFF/GeoJSON + PDF-отчёт, чтобы вставить результат в проектную документацию.
- **Признак готовности:**
  - MapLibre-карта с растровым оверлеем, легендой и шкалой вероятности.
  - Селектор минимум из 3 сценариев пересчитывает слой через `/predict`.
  - Кнопка «Экспорт» отдаёт GeoTIFF, GeoJSON агрегатов и PDF-отчёт с метаданными запуска (версия модели, версия датасета, bbox, идентификатор сценария, дата запуска).
  - Сценарии U1–U3 проходят руками без ошибок на пилотной территории.
- **Риск / неопределённость:** **4/10** — стек фиксирован, но риск §10 «разрастание скоупа фронта» реальный; жёсткий gate на U1–U3, всё остальное — в LATER.

#### IN-5. Объяснимость предсказаний (минимум один метод)

- **Обоснование (метрика discovery):** §6.5 «без объяснимости результаты ИНС-оценки не принимаются предметной аудиторией» + §10 риск «низкая объяснимость → критика на защите» с явной митигацией «включить в MVP, а не в бэклог».
- **User story:** Как специалист муниципалитета (и как оппонент на защите), я хочу для выбранной точки / тайла увидеть, какие признаки сильнее всего повлияли на оценку риска, чтобы доверять цифре и защищать её перед коллегами.
- **Признак готовности:**
  - Для выбранного тайла в UI доступна карта важности признаков: integrated gradients для U-Net либо SHAP-explanation для табличного бейзлайна (минимум один из вариантов).
  - Метод и его ограничения задокументированы в README с одним наглядным примером.
- **Риск / неопределённость:** **5/10** — окончательный выбор метода ещё `К ПРОРАБОТКЕ` (discovery §11.4); техническая реализация любого из вариантов посильна, риск — в интерпретации, а не в коде.

#### Сводка по IN

| # | Фича | Риск |
|---|---|---|
| IN-1 | Пайплайн данных | **8** |
| IN-2 | U-Net + бейзлайн на geo hold-out | **6** |
| IN-3 | Inference REST API | **3** |
| IN-4 | Веб-карта U1–U3 | **4** |

### 2.2 OUT — явно НЕ делаем в MVP

| Что режем | Почему режем (со ссылкой на discovery) |
|---|---|
| Дообучение модели под новый регион через UI | §8.2 — вынесено в развитие; превращает прототип в платформу и ломает фокус «один регион — одна обученная модель». |
| Прогноз во времени / динамика паводка | §4 anti-goal: задача не про распространение волны, а про подверженность; смена постановки требует другой архитектуры и других данных. |
| Оперативное оповещение и интеграция с системами экстренного реагирования | §4 anti-goal; зона ответственности государственных систем, юридически и архитектурно вне нашего контура. |
| Оценка ущерба в денежном выражении | §4 anti-goal; требует данных о стоимости активов, которые сознательно не используются (§5.4). |
| Мультитенантность, аутентификация, ролевая модель | §4 и §8.2; прототип демонстрируется на защите, единственный пользователь — докладчик, auth — чистый overhead. |
| Мобильное приложение и оффлайн-режим | §4 anti-goal; первичная персона работает за десктопом с ГИС-инструментами (§3), мобильный сценарий не покрыт. |
| Интеграция с реальной ливневой канализацией | §8.2; требует закрытых ведомственных данных, что противоречит §5.4. |
| Ансамбли моделей и количественная оценка неопределённости | §8.2; ценно научно, но удваивает время на ML-фазу и не нужно для базового сравнения с бейзлайном (§9.1.1). |
| Развёртывание в продуктивной инфраструктуре, нагрузочное тестирование | §8.2; цель — Docker Compose на одной машине (§7.2), прод-нагрузки нерелевантны для защиты. |
| Real-time данные через сторонние API | §5.4 — явный запрет; ломает воспроизводимость, которая в работе важнее свежести данных. |

---

### 2.3 LATER — после демо (бэклог развития)

1. **Дообучение / файнтюнинг на новый регион через UI** — естественное продолжение, если работа пойдёт в практику муниципалитетов (discovery §8.2).
2. **Ансамбли + калиброванная оценка неопределённости на пиксель** — закрывает методический пробел и усиливает научную часть для публикации (discovery §6.4, §8.2).
3. **Сравнение с альтернативными архитектурами** (CNN, GNN по графу водосбора, гибриды) — discovery §6.2 помечено как `К ПРОРАБОТКЕ`; минимум одна альтернатива нужна для статьи, в MVP — только U-Net.
4. **Расширение на 2–3 дополнительные пилотные территории и кросс-региональная валидация** — усиливает заявку на обобщаемость (discovery §9.1.2).
5. **PDF-отчёт как полноценный экспертный документ** (ссылки на источники данных, версии моделей, рекомендации по интерпретации) — discovery §11.6 ещё не закрыт; в MVP — минимальная версия с метаданными.

## 3. Метрики, по которым считаем MVP сданным

Дублируются здесь как чек-лист приёмки прототипа (свод из discovery §9):

- ✅ IN-1: сборка системы и пересчёт пайплайна на чистой машине ≤ 1 день.
- ✅ IN-2: статистически значимое превосходство U-Net над бейзлайном по IoU / F1 / PR-AUC на geographic hold-out.
- ✅ IN-3: p95 инференса по тайлу ≤ 5 сек на CPU.
- ✅ IN-4: сценарии U1, U2, U3 проходят без ошибок на пилотной территории.
- ✅ IN-5: для произвольного тайла доступна карта важности признаков, метод задокументирован.

## 5. Разделы концепта, которые остаются открытыми

> Эти разделы обязательны для финальной версии `concept.md` (см. discovery §12, шаг 2), но не блокируют переход к `tz.md` по скоупу MVP. Каждый ведётся в отдельной сессии.

1. **Научная новизна** — окончательная формулировка после R&D-фазы 1; пока — заявка из discovery §2.3.
2. **Обзор ближайших аналогов из литературы** — `К ПРОРАБОТКЕ`; обязателен для митигации риска discovery §10 «слабая позиция по новизне».
3. **Окончательный выбор пилотной территории по критериям §5.3** — `К ПРОРАБОТКЕ` (discovery §11.1); до решения в коде используется параметризованный bbox.
4. **Метод объяснимости** (integrated gradients vs SHAP vs оба) — `К ПРОРАБОТКЕ` (discovery §11.4); влияет на признак готовности IN-5.
5. **Параметры преднастроенных метео-сценариев** — `К ПРОРАБОТКЕ` (discovery §11.5); влияет на признак готовности IN-4.
